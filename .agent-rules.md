# Agent Rules For `ciecopilot-home`

## Role of Agent Orchestrator

Use Agent Orchestrator as an execution coordinator only.

- It owns worktree isolation, session routing, PR/CI/review reactions, and worker supervision.
- It does not replace product planning, runtime contract design, or task decomposition judgment.
- If orchestration behavior conflicts with the frozen docs below, the docs win.

## AO Runtime Hygiene

For AO-managed work in this repo:

- Treat live GitHub PR, CI, and review state as canonical when AO lifecycle metadata disagrees with GitHub.
- Before acting on a stale `ci_failed`, `changes_requested`, or `approved-and-green` lifecycle state, reconcile with `ao status -p ciecopilot-home --json`, `gh pr checks`, `gh pr view --json mergeable,reviewDecision,isDraft,url`, and `ao review-check ciecopilot-home --dry-run`.
- For a known PR, prefer repo-local PR-scoped reconciliation via `node scripts/ao-reconcile.js --pr <number> --json --strict` before acting on stale AO lifecycle signals.
- After reconciliation, use `node scripts/ao-doctor.js --pr <number>` or `node scripts/ao-doctor.js` when you need local continuity diagnosis across git/worktree/AO-artifact state.
- After reconciliation and doctor, use `node scripts/ao-lifecycle.js --pr <number> --trigger <trigger>` when you need deterministic next-step reasoning about continue vs restore vs handoff, or about whether to notify the human that a PR appears ready.
- Treat doctor output as diagnose-only. It does not replace phase-1 reconciliation truth and it does not create merge or release authority.
- Treat lifecycle output as decide-only. It does not execute worker restoration, worker handoff, or merge actions.
- Use project-scoped reconciliation only for operator inspection; it is not the sole release gate for this repo.
- If the repository branch, `agent-orchestrator.yaml`, or this rules file changed, or AO-related stash content was restored, stop and recreate the orchestrator session instead of reusing an older one.
- Do not instruct the orchestrator to run repo-local AO control-plane scripts unless those scripts exist in the current branch.

## Engineering Workflow Governance

For default engineering work in this repo:

- The stable local baseline branch is `baseline/origin-main`; treat dated baseline branches or worktrees only as migration-era history, not as the default entry.
- Codex may use the semi-automatic workflow for small, scoped edits: inspect first, edit locally, then clean up before handing control back.
- Start every Codex development turn with `npm run workflow:codex-preflight -- --json` when the workflow scripts exist in the current branch. If Node is not on PATH in the shell, use the repo's known Node runtime and set `TMPDIR=/tmp`.
- Treat preflight output as the workspace contract for the turn:
  - `dirty_requires_classify_stash_commit_or_clean` means classify existing changes before editing and stash or isolate unrelated changes.
  - `protected_branch_requires_codex_or_task_branch_before_commit` means edits may be made, but commits must happen only after creating a `codex/*` or `task/*` branch.
  - `behind_upstream_requires_sync_or_stash_before_work` means do not build long-lived work on a stale protected branch without syncing or intentionally branching first.
- Start new work by syncing baseline with `npm run workflow:baseline:sync` when the workflow scripts exist in the current branch.
- Create task branches and worktrees only through `npm run workflow:task:create -- --id <id> --slug <slug> --base baseline/origin-main`.
- For broad implementation, cross-module work, PR review work, CI repair, or anything needing durable isolation, do implementation work only inside the assigned `task/*` worktree.
- For small edits on `main`, `master`, or `baseline/*`, Codex may edit in place, but must create a `codex/*` or `task/*` branch before committing and must not push protected branches directly.
- Do not continue feature work in the preserved dirty root worktree at `/home/samsen/code/ciecopilot-home`.
- After human merge, close the task with `npm run workflow:task:closeout -- --branch <branch> --confirm closeout` when the local branch still exists.

## Codex Turn Closeout

Before ending a development turn:

- Run focused verification for the touched area, or state the blocker that prevented it.
- Run `git status --short --branch`.
- If changes should be preserved, commit them on a non-protected branch or create a clearly named stash.
- If changes are intentionally left uncommitted, list the exact files and the human decision needed.
- Do not leave unrelated generated files, local tool markers, failed-output files, or accidental scratch files visible in `git status`.
- Final response must include branch name, clean/dirty status, commit or stash identifier when applicable, and verification command outcomes.

## Stage Posture

This repo is in a contract-first runtime foundation stage.

- Quality is more important than speed.
- No fake parallelism.
- No widening of scope beyond the first `9709` runtime slice unless explicitly approved.
- Legacy Study Hub / Learning Path surfaces are compatibility-only and must not regain canonical runtime logic.

## Hard Contract Invariants

These are non-negotiable for this stage:

- `active_scope_bundle` is canonical and typed-ref based.
- Bare `current_question_id` and `current_question_type_id` may exist only as request or `StudySession` summary fields. They must not replace typed refs inside the bundle.
- Questionless sessions are valid. Never invent fake question IDs to satisfy a flow.
- Authoritative scoring is allowed only when all three conditions hold:
  - seeded pilot question type
  - released rubric ref
  - validated uncertainty posture
- Pilot type membership alone must not unlock authoritative scoring.
- Imported or non-pilot questions outside released scope must return explicit `non_released_fallback`, not fake scores or fake point judgements.
- Canonical question family and question type truth must come from seeded registry data, not JS-only constants.
- Canonical-home objects are unique; secondary-topic views are linked references, not duplicated long-lived objects.
- Reconciliation may revise derived state, but `Attempt` and historical `MarkingResult` snapshots remain immutable and lineage must stay visible.
- Artifact lifecycle writes must use explicit intents, not generic `{ state: ... }` payloads.

## Parallelism And Ownership

Parallel work is allowed only when write scopes are truly disjoint.

- Do not split work just because more agents are available.
- Do not create overlapping write scopes.
- Use the ownership rules in `2026-03-20-prd-learning-runtime-pilot-slice-execution.md` verbatim.
- If a task touches files owned by another active worker, stop and escalate or re-sequence the work.

Specific stage guardrails:

- The serial trunk must stabilize before backend and frontend fan-out.
- Frontend runtime work starts only after session API and workspace projection contracts are stable.
- Do not add a fourth backend line if the execution plan does not define one.

## Legacy Boundaries

The following are not acceptable places for new canonical runtime truth:

- `public.ai_tutoring_sessions`
- `public.learning_paths`
- `public.personalized_recommendations`
- `public.study_mode_questions`
- old Study Hub page flow
- old Learning Path page flow

Allowed legacy edits are limited to compatibility reads, redirects, handoff entry points, feature-flagged demotion, and rollout telemetry.

## Task Admission Rules

Before starting a task:

- Confirm the task is explicitly defined in the current execution plan or explicitly approved by the human.
- Confirm the write scope is isolated enough for one worker.
- Confirm the task does not silently widen the stage beyond pilot trigonometry runtime scope.
- Confirm the task does not bypass the frozen schema/API/runtime contract.

If any of those checks fail, stop and ask for direction.

## Verification Rules

Before claiming completion:

- Run focused verification for the touched area.
- Report exact commands and actual outcomes.
- Do not claim success based on code inspection alone.
- Merge authority is owned by live GitHub state plus explicit human approval for AO-managed work in this repo.
- Before treating a PR as release-ready, verify green required checks, no unresolved review threads, and no merge conflicts from GitHub's live state.
- The `approved-and-green` reaction may notify the orchestrator, but it must not bypass live GitHub verification with direct auto-merge.
- Surface blockers, contract ambiguity, and failed verification immediately.

Minimum expectation:

- backend work runs the targeted Jest or API suites for the touched slice
- frontend work runs the targeted tests and any required build checks
- schema or contract work verifies the relevant migration or contract tests

## PR Discipline

- Keep each worker branch narrow and aligned to a single task or tightly related task block.
- Do not mix serial trunk work with downstream fan-out work in one PR unless the execution plan explicitly requires it.
- Prefer smaller contract-aligned PRs over broad refactors.
- If a reviewer comment would violate the frozen contract, escalate instead of implementing it blindly.
