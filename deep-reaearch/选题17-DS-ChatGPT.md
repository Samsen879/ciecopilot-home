Topic Leakage 攻击面分析

Topic Leakage 指 AI 辅导答案涉及当前 topic_path 子树之外的知识点，可能导致“超纲指导”。其攻击面可系统分为：

数据层风险：知识库中 knowledge_chunks.topic_path 标注错误（如标签误分类、空值、格式不符）会使 SQL 过滤失效。研究表明，KGQA 数据集常存在“标注不准确、不完整”及“知识过时或不一致”的问题
1
，提示数据质量问题会导致检索结果错误。错误的 topic_path 可能使子树过滤无效，异常记录被检索出。

检索层风险：现有机制通过 Postgres ltree 子树查询强制过滤，但仍有边缘案例。若 topic_path 值为 NULL 或格式非法，SQL 比较将返回 NULL 而非真
2
，可能绕过查询判断。此外，如果数据库查询参数缺失或注入攻击破坏了过滤条件，也会导致超纲内容被检索。

生成层风险：LLM 的参数化知识（训练数据记忆）可能绕过检索上下文，自主生成超纲内容。LLM 容易产生“hallucination”（幻觉），即回答中包含未出现在检索结果或事实中的信息
3
4
。例如，AWS 指出即使有 RAG，AI 仍存在幻觉风险：“即使具备 RAG 功能，AI 生成不准确信息的问题仍非常严重”
4
。这意味着仅依赖检索结果无法完全约束模型，必须额外检测生成回答。

Prompt 注入攻击：恶意用户可构造输入，将有害指令伪装成用户提问插入到系统提示中，从而劫持模型行为。IBM 研究指出，“Prompt 注入是一种攻击，黑客将恶意内容伪装成用户输入，写入能覆盖 LLM 系统提示的提示语，从而将应用程序变为攻击者的工具”
5
。例如，远程推特 bot 的例子中，用户提交的推文里包含“ignore all previous instructions”之类指令，使模型执行了超纲要求
5
。在 RAG 系统中，提示注入可能诱使 LLM 忽略系统设定的子树约束，输出不相关内容。

边界模糊性：某些知识点在学科体系中互为前置和后置关系，如 A-Level 课程中“微分”（9709.P1）为“积分”（9709.P3）的先修知识。导师在讲解当前题目时引用先修内容是合理的，但难以用简单子树过滤判断。区分“合法的先修引用”与“非法超纲输出”本身是一个设计问题，需要额外的业务规则来约束。

综上，Topic Leakage 可能源自数据标注错误、检索过滤失效、LLM 自主生成以及提示注入。专业安全指南建议，不仅在推理阶段过滤模型输出，也要在数据摄取阶段严格筛查外部知识库内容
6
。

多层防御架构设计

针对以上攻击面，我们设计“多层防御”架构：

Layer 1 – 数据库过滤（已有）：在 SQL 检索时强制按当前 topic_path 子树过滤，如 WHERE topic_path <@ current_topic_path。这一层已部署。还可增强摄取链路安全：在文档入库时进行静态扫描或用 LLM 过滤，检测并剔除隐藏的恶意提示或不相关内容。AWS 建议在摄取流水线中加入过滤器，例如用专门的 LLM 检测文件中的不良序列或不相关上下文
7
，减少注入风险。

Layer 2 – Prompt 约束：设计严格的系统提示，引导模型仅引用当前子树相关内容。例如，系统提示可声明「请只基于当前主题路径下的教材内容回答，不要提供超出范围的知识」。尽管系统提示是软约束，LLM 可能会遗忘或忽略，但它仍是第一道防线。应尽量清晰、明确地说明禁止超纲输出并要求模型引用证据。

Layer 3 – 输出后检测：对 LLM 生成的回答进行自动检测，若判断为超纲则拒答。比较三种方案：

方案A：NER+知识图匹配。对回答进行命名实体识别（NER），提取关键概念词，然后查询它们在教学知识图中的主题路径，检查是否都在当前子树内。GraphRAG 等方法即通过 LLM 提取实体和关系来构建知识图
8
。示例代码：
python
复制
# 方案A示例（伪代码）：提取回答中的实体并检验是否超纲
answer = "ANSWER_TEXT"
# 假设有函数 extract_entities(answer) 提取实体列表
entities = extract_entities(answer)  
allowed_topics = get_subtree_topics(current_topic_path)  # 获取子树下允许的概念集合
for ent in entities:
    if ent not in allowed_topics:
        leakage_flag = True
        break

如果检测到不在允许子树内的实体，即标记为泄露。这种方法需要维护一个主题-概念的知识库。优势是针对概念准确；缺点是依赖 NER/知识库的准确性，可能漏检概念或误识。
方案B：语义嵌入相似度。将回答与当前主题上下文分别编码成向量，计算余弦相似度：若回答中某些句子与上下文相似度很低，说明其信息可能不源自已检索到的内容
9
。AWS 示例代码使用 Titan Embedding 模型计算相似度
9
。示例代码：
python
复制
from sentence_transformers import SentenceTransformer, util
model = SentenceTransformer('all-MiniLM-L6-v2')
context_emb = model.encode(context_text)
answer_emb = model.encode(answer_text)
similarity = util.cos_sim(context_emb, answer_emb)[0][0]
# 假设阈值设为0.7，如果相似度过低则视为疑似超纲
if similarity < 0.7:
    leakage_flag = True

根据 AWS 实验，嵌入相似度检测在某数据集上精度约0.48、召回0.90，延迟非常低
10
（见下表）。这种方法实现简单、速度快（只需一次向量计算），但精度中等；需要调参设阈值以平衡误报率。
方案C：LLM 评审（LLM-as-Judge）。调用一个独立的 LLM（可与主模型相同或不同）作为“审判”，判断回答是否超纲。即向 LLM 提供用户问题、回答与当前主题信息，让其输出「是否包含超纲内容」的判断。NBD 研究表明，LLM作为评审可以有效检测输出问题
11
。示例代码（伪）如下：
python
复制
import openai
prompt = f"""System: 请判断以下回答是否包含超出当前主题范围的信息。
Context: {context}
Answer: {answer}
Question: 是否有任何答案中的内容在当前主题子树之外？请回答是或否。"""
judge_response = openai.ChatCompletion.create(
    model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=0
)
verdict = judge_response.choices[0].message.content
if "是" in verdict:
    leakage_flag = True

根据 AWS 报告，LLM 评审检测在某数据集上可达到精度0.75、召回0.94，但延迟较高（约0.53秒）
10
。它对复杂语义有较强判断能力，但成本（API调用延迟）也最高。

三种方案各有优劣：NER+KG 方法精度较低但可明确定位违规概念；相似度方法快速但可能遗漏精细语义；LLM评审精度高但代价高。可以综合使用：例如先用快速的相似度或简单规则过滤明显违规，再用 LLM 判断难以判定的案例

Layer 4 – 异步审计：离线或后台定期审查所有答复的证据路径，确认返回的 evidence[] 均在子树内。一旦发现泄露，会触发报警并修正过滤策略。这类似安全监控，对话日志可以被用作审计追溯。参考 AWS 安全最佳实践，应对未被实时检测到的恶意或不一致内容进行额外审查

合法边界处理规则

在 A-Level 教学场景中，允许参考先修知识，但需明确定义规则。例如，可规定只允许引用当前主题节点向上的先修概念（沿 PREREQUISITE_OF 边），而不允许引用平级或无关节点。可以设置“允许上溯深度”阈值，如只允许引用一层或两层之内的直接先修知识。举例：当前在 9709.P3.Integration，允许参考其先修 9709.P2.Differentiation，但不允许引用 9709.P4（高级积分）之外的内容。该规则保证正当补充基础知识，同时防止越级讲解。实现时可在知识图中对每个主题标注先修关系，上溯时检查路径中是否只沿着先修链移动，即可区分合法引用与非法越界。

评测基准设计

为了量化检测器性能，应构建 Topic Leakage 检测基准库，包括：明显越界案例、微妙越界案例、合法先修引用案例、正常回答案例，每类各30个以上。可以借鉴现有研究方法，如使用 LLM 生成或人工编写各类题干和答案。例如，ELOQ 数据集通过 LLM+人工生成了种类丰富的“超范围”问题
14
，可模拟类似思路构造测试样本。评估指标重点选择 F1 分数，但偏向高召回（漏报要最低）。也可分项测试：NER+KG 方法精度低但应确保召回高，相似度方法快速高召回，LLM 评审高精度高召回，但延迟大。最终应报告各方法在基准上的 Precision/Recall/F1，以及处理时延，以指导实际部署选择。

线上监控指标

需实时统计 Topic Leakage 率，并在 Grafana 等监控系统中可视化：定义指标如总回答数 (denominator) 与被标记为超纲回答数 (numerator)，计算泄露率 (例如每分钟或每小时)。可设计仪表盘显示总回答量、泄露次数、泄露率曲线及主题分布等。报警阈值可设为泄露率超过 PRD 要求（如 1%）时触发。结合 AWS 推荐的可观测实践，监控回答契约是否被满足（topic_leakage_flag 为 false）等辅助指标，以确保系统符合安全标准
6
